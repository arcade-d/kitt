# Spec — Séquence de boot K2000 « K.I.T.T. SYSTEMS ONLINE »

Date : 2026-06-13
Statut : design validé (brainstorming), prêt pour plan.

## Contexte

Au lancement, `main` → `KittApp` (thème K2000) → `BootstrapGate`. Ce dernier
watche `modelsReadyProvider` : en état `loading` (init `ModelManager` + extraction
des voix embarquées), il n'affiche qu'un `CircularProgressIndicator` rouge nu, puis
bascule vers `CompanionScreen` (prêt) ou `ModelDownloadScreen` (LLM à télécharger).

On remplace ce spinner par un **rituel de boot K2000** cohérent avec l'écran CRT
(suite du chantier [[k2000-crt-screen-chantier]] ; déjà reporté comme suivi).

## Décisions (brainstorming)

- **Déclenchement** : durée minimale ~2,5 s **+ couvre l'init**. Le boot reste
  affiché tant que `modelsReady` n'est pas résolu (data/erreur) ; il dure au moins
  2,5 s même si l'init est instantanée. Jamais de spinner nu.
- **Interruptible** : **tap pour passer** — saute le reste **uniquement si l'init
  est résolue** (sinon on continue d'attendre, frame « ONLINE » tenue).
- **Contenu** : rituel complet — power-up CRT → lignes de check qui défilent →
  scanner qui monte en régime → « K.I.T.T. SYSTEMS ONLINE ».

## Objectifs

1. Boot K2000 immersif à chaque lancement, qui masque proprement l'init.
2. Aucune régression de l'aiguillage existant (Companion / Download / erreur).
3. Perf : animation légère (`CustomPainter` + `AnimationController`), pas d'asset.

## Non-objectifs

- Son de boot.
- Lignes de check reflétant le **vrai** statut des modèles (cosmétiques en v1).
- Toucher au pipeline, aux ports ou aux écrans Companion/Download.

## Architecture

### `BootScreen` (`lib/ui/boot_screen.dart`)
Widget animé autonome, `StatefulWidget` + `SingleTickerProviderStateMixin`.
- Un `AnimationController` de ~2,5 s, joué une fois puis maintenu sur la dernière
  frame (« ONLINE ») si on attend encore l'init.
- Rendu via `CustomPainter` (CRT : lueur ambre, scanlines, scanner rouge qui monte)
  + texte `KittText` pour les lignes de check et le titre.
- Timeline (fractions de la durée) :
  1. **0,00–0,15** : power-up — flash bref + scanlines qui se stabilisent.
  2. **0,15–0,72** : lignes de check apparaissant l'une après l'autre —
     `SCANNER……OK`, `VOIX……OK`, `MÉMOIRE……OK`, `CERVEAU……` (ambre/rouge).
  3. **0,60–1,00** : scanner Larson qui monte en régime + `K.I.T.T. SYSTEMS ONLINE`
     en gros ambre.
- `onSkip` : callback invoqué sur tap (`GestureDetector`). Le `BootScreen` ne décide
  pas de la transition ; il signale juste le tap.
- Couleurs/typo via `KittColors`/`KittText`.

### `BootstrapGate` (`lib/ui/bootstrap_gate.dart`) → `ConsumerStatefulWidget`
Orchestration boot ↔ résultat :
- `initState` : démarre un `Timer(2.5s)` → `setState(_minElapsed = true)`.
- État local : `_minElapsed`, `_skipRequested`.
- `build` : `final modelsReady = ref.watch(modelsReadyProvider);`
  `final initResolved = !modelsReady.isLoading;`
  `final showBoot = shouldShowBoot(minElapsed: _minElapsed, skipRequested: _skipRequested, initResolved: initResolved);`
- Si `showBoot` → `BootScreen(onSkip: () => setState(() => _skipRequested = true))`.
- Sinon → aiguillage existant `modelsReady.when(loading/error/data)` **inchangé**
  (le `loading` n'arrive en pratique plus puisqu'on ne sort du boot qu'une fois
  `initResolved`, mais on le garde par sûreté).
- `dispose` : annuler le `Timer`.

### Fonction pure `shouldShowBoot` (`lib/ui/boot_gate_logic.dart`)
```dart
bool shouldShowBoot({
  required bool minElapsed,
  required bool skipRequested,
  required bool initResolved,
}) =>
    !(initResolved && (minElapsed || skipRequested));
```
Testable en isolation (cœur de la logique de transition).

## Flux

```
launch → BootstrapGate(initState: Timer 2.5s)
  ├─ showBoot == true  → BootScreen (anime ; tap → _skipRequested)
  └─ showBoot == false → modelsReady.when → Companion / Download / erreur
transition quand : initResolved && (minElapsed || skipRequested)
```

## Gestion d'erreurs / dégradations

- Si `modelsReady` est en **erreur** : `initResolved == true` → on sort du boot
  après 2,5 s (ou tap) vers l'écran d'erreur existant (inchangé).
- Mode mock : `modelsReady` data quasi instantané → le boot dure tout de même 2,5 s.
- Aucune dépendance réseau ; aucun asset.

## Tests

- **Unitaire** `shouldShowBoot` : table de vérité (init non résolue → toujours boot ;
  init résolue + minElapsed → non ; init résolue + skip → non ; init résolue sans
  min ni skip → boot).
- L'animation (`BootScreen`) est visuelle → validée par `analyze` + build + device.

## Questions ouvertes

- Police 7-seg dédiée (asset) vs mono actuel — cosmétique, hors v1.
- Lignes de check reflétant le vrai statut (STT/TTS/LLM) — amélioration ultérieure.
